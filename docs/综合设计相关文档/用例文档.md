## 3. 用例详细规约

### UC01 — 注册账号

| 项目 | 内容 |
|------|------|
| **编号** | UC01 |
| **名称** | 注册账号 |
| **参与者** | 学生（ACT-1）、教师（ACT-2） |
| **前置条件** | 未登录；学号/工号未被占用 |
| **后置条件** | 用户记录写入 `users` 表，角色由系统自动判定：学生学号→student；教师工号需管理员后台导入预设；admin 工号由数据库初始化创建 |

**基本流程**：
1. 用户点击"注册"按钮，进入注册页面
2. 用户输入：学号/工号（username）、密码
3. 系统校验：学号/工号唯一性、密码强度（≥6 位）
4. 系统创建用户记录，角色默认 `student`，密码经 bcrypt 哈希存储
5. 系统跳转至登录页，提示"注册成功，请登录"
6. 用户首次登录后可补充真实姓名（real_name）和邮箱（email）
7. **用例结束**

**备选流程**：
- **2a. 学号/工号已存在**：系统提示"该学号/工号已被注册"；返回步骤 2
- **2b. 密码强度不足**：系统提示"密码至少 6 位"；返回步骤 2

**补充说明**：注册接口不暴露角色选择——教师由管理员在后台将指定工号预设为 teacher 角色后通知教师登录；admin 账号通过数据库初始化脚本直接 INSERT（role='admin'），同样使用工号。

### UC02 — 登录系统

| 项目 | 内容 |
|------|------|
| **编号** | UC02 |
| **名称** | 登录系统 |
| **参与者** | 学生（ACT-1）、教师（ACT-2）、管理员（ACT-3） |
| **前置条件** | 已注册账号且未被禁用 |
| **后置条件** | 用户获得 JWT access_token（1h）+ refresh_token（7d），跳转至对应角色主页 |

**基本流程**：
1. 用户打开登录页，输入学号/工号和密码
2. 系统查询 `users` 表，bcrypt 验证密码
3. 系统生成 JWT access_token（HS256，1h 有效）和 refresh_token（7d 有效）
4. 系统根据用户 role 跳转至对应主页：student → 学生门户、teacher → 教师门户、admin → 管理后台
5. 系统记录登录日志（audit_logs）
6. **用例结束**

**备选流程**：
- **2a. 用户名不存在**：系统提示"用户名或密码错误"；返回步骤 1
- **2b. 密码错误**：系统提示"用户名或密码错误"；返回步骤 1
- **2c. 账号已被禁用**：系统检测到 `is_active=FALSE`，提示"账号已被禁用，请联系管理员"；**用例终止**
- **2d. 需强制改密**：系统检测到 `force_password_change=TRUE`，返回 `require_password_change: true`，前端强制跳转修改密码页
- **2e. Token 即将过期（剩余 < 30min）**：前端自动调用 `POST /auth/refresh` 刷新

### UC03 — 关键词检索

| 项目 | 内容 |
|------|------|
| **编号** | UC03 |
| **名称** | 关键词检索 |
| **参与者** | 学生（ACT-1）、教师（ACT-2） |
| **前置条件** | 已登录 |
| **后置条件** | 无 |

**基本流程**：
1. 用户在搜索框输入关键词（如"栈的实现"）
2. 用户可选限定筛选条件：课程、文件类型
3. 系统对 `documents.title`, `documents.description`, `chunks.content` 执行 ILIKE 模糊匹配（仅搜索 status='approved' 的文档）
4. 系统按文档标题匹配度排序，分页返回结果
5. 结果列表展示：文档标题、所属课程、匹配内容片段（关键词高亮）、文件类型
6. **用例结束**

**备选流程**：
- **3a. 无匹配结果**：系统提示"未找到相关文档，请尝试其他关键词"
- **3b. 关键词为空**：系统提示"请输入搜索关键词"

**业务规则**：
- 仅返回审核通过的文档（status='approved'）
- 每页默认 10 条，最多 50 条
- 中文采用 ILIKE 模糊匹配（PostgreSQL `simple` 分词对中文无效，故不采用全文检索索引）

### UC04 — 语义问答（RAG 智能问答）

| 项目 | 内容 |
|------|------|
| **编号** | UC04 |
| **名称** | 语义问答 |
| **参与者** | 学生（ACT-1）、教师（ACT-2） |
| **前置条件** | 已登录；系统中存在已入库的课程文档 |
| **后置条件** | 问答记录写入 `qa_history` 表 |
| **包含用例** | UC17（提示注入防护） |
| **扩展用例** | UC18（无答案拒答） |

**基本流程**：
1. 用户输入自然语言问题（如"AVL 树的旋转操作有哪几种？"），可选限定课程
2. 系统执行输入安全检查（`UC17 提示注入防护`）
3. 系统并行执行双路检索：
   - 语义检索：问题向量化 → ChromaDB 余弦相似度检索 Top-10
   - 关键词检索：问题关键词 → ILIKE 模糊匹配 Top-5
4. 系统合并去重，按分数降序排序
5. 系统相关性过滤：余弦相似度 < 0.45 的片段淘汰
6. 若全部淘汰 → 触发 `UC18 无答案拒答`
7. 系统构建 Prompt：System Prompt + Top-K 参考资料 + 用户问题
8. 系统调用 `deepseek-v4-flash` 生成答案
9. 系统输出校验：检测答案是否包含引用标注 `[1]`, `[2]`
10. 系统展示答案（Markdown 渲染）+ 引用来源列表
11. 系统记录问答历史（user_id, question, answer, sources, course_id, latency_ms）
12. **用例结束**

**备选流程**：
- **2a. 检测到提示注入**：系统拒绝处理，返回"检测到不安全的输入，请重新提问"
- **2b. 用户输入超长（>2000 字符）**：系统截断为前 2000 字符，继续处理
- **6a. 全部片段低于阈值**：见 `UC18 无答案拒答`
- **9a. 答案不含引用标注**：系统判定为异常，触发拒答

**业务规则**：
- RAG 管线采用 LangChain LCEL 编排（`ChatPromptTemplate → ChatDeepSeek → StrOutputParser`）
- LLM 参数：temperature=0.3，max_tokens=1024
- 上下文窗口上限：3000 字符

### UC06 — 提交反馈

| 项目 | 内容 |
|------|------|
| **编号** | UC06 |
| **名称** | 提交反馈 |
| **参与者** | 学生（ACT-1）、教师（ACT-2） |
| **前置条件** | 已登录；存在问答记录 |
| **后置条件** | 反馈记录写入 `feedback` 表 |

**基本流程**：
1. 用户在问答详情页底部看到三个反馈按钮：👍 有用 / 👎 无用 / ❌ 错误

- **3c. 撤销反馈**：用户删除反馈记录，恢复未反馈状态

**补充说明**：反馈数据将被记录到 `feedback` 表，管理端可通过统计接口查看各类型占比，持续优化检索与生成质量。

### UC08 — 上传课程资料

| 项目 | 内容 |
|------|------|
| **编号** | UC08 |
| **名称** | 上传课程资料 |
| **参与者** | 教师（ACT-2） |
| **前置条件** | 已登录，角色为 teacher；课程已创建 |
| **后置条件** | 文档记录写入 `documents` 表（status=pending），等待审核 |
| **包含用例** | UC12a（数据脱敏检测） |
| **扩展用例** | UC12（审核文档） |

**基本流程**：
1. 教师点击"上传资料"
2. 选择文件（PDF/DOCX/TXT/PPTX/MD），填写元数据：课程、文件类型、标题、描述、标签
3. 系统校验文件类型白名单、MIME、大小（≤50MB），计算 SHA-256 去重
4. 系统执行数据脱敏检测（`UC12a`）
5. 系统保存文件到 `data/files/`，创建文档记录（status='pending', processing_status='pending'）
6. 系统提示"上传成功，请等待管理员审核"
7. **用例结束**

**备选流程**：
- **2a. 文件类型不支持**：系统提示"仅支持 PDF/DOCX/TXT/PPTX/MD 格式"
- **2b. 文件超过 50MB**：系统提示"文件过大，最大支持 50MB"
- **2c. 文件已存在（SHA-256 重复）**：系统提示"该文件已上传过"
- **3a. 检测到敏感信息**：系统提示"文档中检测到可能的个人信息（手机号/身份证号），建议脱敏后重新上传"

### UC12 — 审核文档

| 项目 | 内容 |
|------|------|
| **编号** | UC12 |
| **名称** | 审核文档 |
| **参与者** | 管理员（ACT-3） |
| **前置条件** | 已登录，角色为 admin；存在待审核文档 |
| **后置条件** | 文档状态变更；审核通过后触发自动解析入库 |
| **扩展用例** | UC16（文档解析与向量化入库） |

**基本流程**：
1. 管理员进入"待审核文档"列表
2. 点击某文档查看详情：文件元数据、上传者、可预览文件内容
3. 管理员点击"通过"
4. 系统更新文档 status='approved'
5. 系统触发后台异步任务：文档解析 → chunk 切分 → 向量化 → ChromaDB 入库（`UC16`）
6. 系统记录审核日志（audit_logs）
7. **用例结束**

**备选流程**：
- **3a. 驳回**：管理员选择"驳回"，填写驳回原因
- 系统更新 status='rejected'，填写 audit_comment
- 上传者可在文档管理中看到驳回原因

**业务规则**：审核通过后才触发解析和向量化入库，避免未审核或驳回文档浪费计算资源。

### UC15 — 管理用户账号

| 项目 | 内容 |
|------|------|
| **编号** | UC15 |
| **名称** | 管理用户账号 |
| **参与者** | 管理员（ACT-3） |
| **前置条件** | 已登录，角色为 admin |
| **后置条件** | 用户账号状态变更 |

**基本流程 — 重置密码**：
1. 管理员进入用户列表，搜索目标用户（按学号/工号）
2. 点击"重置密码"
3. 系统生成随机临时密码（8 位），bcrypt 哈希后更新 `password_hash`
4. 系统返回临时密码，管理员告知用户
5. 用户下次登录时强制跳转修改密码页

**基本流程 — 禁用/启用账号**：
1. 管理员在用户列表中找到目标用户
2. 点击"禁用"按钮，系统弹出二次确认
3. 确认后系统设置 `is_active=FALSE`
4. 该用户无法登录（登录时校验 is_active），已有 token 失效
5. 管理员可随时"启用"恢复账号

**基本流程 — 变更角色**：
1. 管理员在用户列表中找到目标用户
2. 将角色从 student 变更为 teacher（或反向）
3. 系统更新 `role` 字段
4. 用户下次登录时获得新角色权限

**业务规则**：admin 角色不可通过此接口变更（仅数据库直操作）。密码重置需记录审计日志。

### UC16 — 文档解析与向量化入库

| 项目 | 内容 |
|------|------|
| **编号** | UC16 |
| **名称** | 文档解析与向量化入库 |
| **参与者** | 系统（ACT-4） |
| **触发条件** | 管理员审核通过文档后自动触发 |
| **前置条件** | 文档 status='approved' |
| **后置条件** | 文档 processing_status='completed'；chunks 写入 PostgreSQL；向量写入 ChromaDB |

**基本流程**：
1. 系统更新文档 processing_status='processing'
2. Docling 引擎解析文档（PDF/DOCX/PPTX → 纯文本；若失败则 PyMuPDF 兜底）
3. RecursiveCharacterTextSplitter 切分：chunk_size=800, overlap=150
4. BAAI/bge-large-zh-v1.5 模型逐批向量化（batch_size=32）
5. Chunk 元数据写入 PostgreSQL `chunks` 表
6. Chunk 向量 + 内容写入 ChromaDB（`hnsw:space=cosine`）
7. 系统更新 documents.chunk_count, documents.processing_status='completed'
8. **用例结束**

**备选流程**：
- **2a. 解析失败**：标记 processing_status='failed'，记录错误日志

| **前置条件** | 用户提交问题文本 |

### UC18 — 无答案拒答

|------|------|

| **参与者** | 系统（ACT-4） |

4. **用例结束**

**备选流程**：
- **1a. 问题与课程无关**（如天气、闲聊）：System Prompt 规则 5 要求 LLM 回复"这是课程学习助手，请提出与课程相关的问题。"
