#### 1.3.2 RAG 问答流程（LangGraph 编排）

基于 LangGraph 的有状态图编排，RAG 问答被建模为 5 个业务 Node + 3 条条件路由 + 2 个终点（reject / return_answer）。

**Graph 拓扑图**：

```
                        START
                          │
                          ▼
                    classify_intent ────→ [非NORMAL] → REJECT(意图拦截)
                          │
                          ▼
                      rag_search
                          │
              ┌───────────┴───────────┐
              │ 有结果                 │ 无结果
              ▼                       ▼
        build_context          route_after_rag
              │                   ┌───┴───┐
              │              web_on     web_off
              │                   │         │
              │                   ▼         ▼
              │             web_search    REJECT
              │              ┌───┴───┐  (无结果拒答)
              │           成功    失败/API挂
              │              │         │
              │              ▼         ▼
              │           (继续)    REJECT
              │                   (搜索失败拒答)
              └──────┬────────┘
                     ▼
              generate_answer
                     │
                     ▼
              review_output
                     │
              ┌──────┴──────┐
              │ PASS        │ REJECT
              ▼             ▼
        RETURN_ANSWER    REJECT
                       (输出违规拒答)
```

**5 个业务 Node 职责**：

| Node | 类型 | 输入 → 输出 | 说明 |
|------|------|------------|------|
| `classify_intent` | LLM | `question` → `intent` (NORMAL/CHEATING/SENSITIVE/ATTACK) | 跨语言（简中/英文/文言文）一致性分类，temperature=0 |
| `rag_search` | 函数 | `question`, `course_id` → `internal_results[]`, `has_internal_results` | 向量检索 + 阈值 0.45 过滤 |
| `build_context` | 函数 | `internal_results` → `context`, `sources[]`, `search_mode="internal"` | 拼接 [来源N: 标题] 格式 + 构造结构化 sources |
| `web_search` | API+函数 | `question` → `context`, `sources[]`, `has_web_results`, `search_mode="web"` | Tavily API 搜索，含超时/异常安全降级 |
| `generate_answer` | LLM | `context`, `question`, `chat_history` → `answer` (含 [来源N]) | DeepSeek 生成，区分事实性提问与作业索要 |
| `review_output` | 函数+LLM | `answer`, `sources` → `review_result`, `matched_sources[]` | 机械引用校验 + LLM 合规审查 |

**3 条条件路由**：

```python
route_after_intent(state) → "rag_search" | "reject"
route_after_rag(state)    → "build_context" | "web_search" | "reject"
route_after_web(state)    → "generate_answer" | "reject"
```

**多轮对话支持**：State 中 `chat_history: Annotated[list, operator.add]` 字段跨 turn 累积。`generate_answer` 的 Prompt 包含格式化后的对话历史。Graph 编译时附带 `AsyncPostgresSaver` checkpointer，同一 `thread_id` 的多次调用自动恢复上轮对话状态。

**持久化方案**：
- Graph 状态通过 checkpointer 持久化，按 `thread_id` 组织
- 业务层 `user_sessions` 表作为轻量索引（`thread_id`, `user_id`, `first_question`, `turn_count`）
- 历史查询直接通过 `graph.aget_state_history(thread_id)` 读取，过滤 `snapshot.next == ()` 取终端状态

### 1.4 异步任务与消息队列

**选型**: 使用 **FastAPI `BackgroundTasks`** 作为文档异步处理的轻量方案，不引入 Celery 等重量级任务队列。

| 考量维度 | FastAPI BackgroundTasks | Celery + Redis |
|----------|------------------------|----------------|
| 复杂度 | 零配置，一行代码添加任务 | 需部署 Worker + Broker + Result Backend |
| 适用场景 | 轻量异步（文档解析 ~5-30s） | 长时间任务、定时任务、分布式 |
| 本项目需求 | ✅ 文档上传后异步解析即可 | ❌ 过度设计，课程项目不需要 |
| 可靠性 | 进程内执行，进程重启任务丢失 | 持久化队列，支持重试 |
| 扩展性 | 单进程，并发受限 | 多 Worker 横向扩展 |

**处理流水线**：

```
1. POST /documents/upload → 保存文件 → 写入 DB (processing='pending') → add_background_task(process_document, doc_id) → 立即返回 202
2. Background Task 异步执行:
   a. UPDATE processing='processing'
   b. Docling 解析 → Markdown
   c. Chunker 切分 → [{index, content}]
   d. INSERT chunks (批量 100 条/次)
   e. EmbeddingService.embed_documents(texts) → 向量列表
   f. ChromaDB.add(ids, embeddings, metadatas) → 写入向量库
   g. UPDATE processing='completed', chunk_count=N
3. 异常处理:
   a. 任一步骤抛异常 → UPDATE processing='failed'
   b. 写入 audit_logs (action='doc_process_failed', detail={error, traceback})
   c. 管理员可在后台查看失败原因，通过 POST /documents/{id}/reprocess 重试
```

**未来演进路径**: 若后续需要定时任务（过期 token 清理、日志归档），可通过 `asyncio.create_task` + `apscheduler` 实现进程内调度。如需分布式部署，将 `BackgroundTasks` 替换为 Redis + ARQ/RQ 任务队列。

### 1.5 容错与弹性设计

| 故障场景 | 影响范围 | 应对策略 |
|----------|----------|----------|
| **LLM API 超时/不可用** (DeepSeek) | 问答功能 | 降级为仅返回检索结果（chunks），提示"AI 回答暂不可用"；调用设置 `timeout=15s, max_retries=1`，2 次失败后降级 |
| **Embedding API 不可用** (SiliconFlow) | 全站检索/问答 | FastAPI `startup` 事件验证 API 连通性；运行时异常自动降级到本地 `bge-large-zh-v1.5` 模型（如已安装）；均不可用时 fallback 到纯 ILIKE 关键词检索 |
| **PostgreSQL 连接断开** | 全站 | SQLAlchemy `pool_pre_ping=True` 自动检测断开并重连；连接池 `pool_size=20, max_overflow=10`；超时 `pool_recycle=3600` |
| **ChromaDB 数据损坏** | 向量检索 | 定期备份 `data/chroma` 目录（每日 rsync）；提供 `POST /admin/rebuild-index` 从 chunks 表全量重建向量 |
| **磁盘空间不足** | 文件上传 | 上传前 `shutil.disk_usage()` 检查可用空间 ≥ 1GB；超限时返回 507 "存储空间不足"；loguru 告警 |
| **大文件上传中断 (>50MB)** | 该次上传 | 前端分片上传（5MB/chunk），后端 `/upload/chunk` 临时存储片段，全部完成后 merge 并校验 SHA-256；超时片段自动清理 |
| **DeepSeek API 返回敏感/违规内容** | 问答内容安全 | 三层防护体系：① LLM 意图分类（classify_intent, 4 类标签） ② System Prompt 约束（区分事实性提问 vs 作业索要） ③ LLM 输出审查（review_output, PASS/REJECT） |
| **并发问答请求堆积** | 问答延迟增加 | LLM 调用使用 `asyncio.Semaphore(3)` 限制最大并发；超出排队返回 `202 Accepted` + `Retry-After` 头，前端轮询结果 |
| **refresh_token 泄露** | 账号安全 | 每次 refresh 旋转旧 token（旧 token 立即 revoked）；登录时若检测到异常 IP/设备，吊销该用户所有 refresh_token |

```bash
#### 2.4.5 DeepSeek V4 Flash 模型约束与应对

`deepseek-v4-flash` 为**推理模型（thinking mode）**，存在以下关键约束需要在所有 LLM 调用点遵循：

| 约束 | 说明 | 应对 |
|------|------|------|
| **不支持 tool calling** | `with_structured_output()` 报错："Thinking mode does not support this tool_choice" | 使用 `response_format={"type": "json_object"}` 替代 |
| **显式 max_tokens 陷阱** | 设置 `max_tokens=N` 时，所有 N 个 token 被 thinking 消耗，可见输出为空 | 所有 LLM 调用**不传 `max_tokens`**（默认 None），让模型自行管理 |
| **支持 JSON Mode** | `response_format={"type": "json_object"}` 正常工作，Prompt 中须包含 "json" 字样 | `generate_answer` 节点使用 JSON Mode 强制结构化输出 |

**各节点 LLM 调用参数规范**：

| 节点 | temperature | max_tokens | 其他 |
|------|:---:|:---:|------|
| `classify_intent` | 0 | 不传 | 纯文本分类 |
| `generate_answer` | 0 | 不传 | `model_kwargs={"response_format": {"type": "json_object"}}` |
| `review_output` | 0 | 不传 | 语义审查，仅回复 PASS / REJECT |

> **历史教训**: 最初 `classify_intent` 和 `review_output` 均设置了 `max_tokens=10/5`，导致这两个节点在 deepseek-v4-flash 上静默失败（返回空字符串），分类降级为 NORMAL、审查降级为 PASS，安全防护完全旁路。修复后已验证两个节点正常工作。

# Embedding — 本地模型（仅 EMBEDDING_PROVIDER=local 时需要）
uv pip install sentence-transformers torch --index-url https://download.pytorch.org/whl/cu121  # bge-large-zh-v1.5

# 可选
uv pip install langgraph                         # 如需状态机流程增强（v1 暂不启用）
```

> **v1 注意事项**：Python 需 ≥3.10（我们 3.12 满足）。`langchain` 包 v1 收敛为 agent 核心，`ChatPromptTemplate`、`StrOutputParser` 等基础组件在 `langchain-core` 中保留；文本分割器独立为 `langchain-text-splitters`；遗留 chains/retrievers 如需使用需安装 `langchain-classic`。
>
> **版本锁定**：LangChain 生态变化极快，实际开发前应在 `pyproject.toml` 中锁定精确版本号（如 `langchain==1.3.4`），并通过 `uv.lock` 固定整个依赖树。`langchain-docling` 包需确认 Docling 官方是否已发布独立的 LangChain 集成包，如未发布则改用 Docling 原生 API + `BaseLoader` 自定义加载器。

- **布局策略**：移动端侧边栏收起为汉堡菜单（drawer），桌面端固定左侧
- **关键页面适配**：
  - 搜索页：移动端搜索框全宽、结果卡片单列；桌面端双列布局
  - 文档列表：移动端卡片视图，桌面端表格视图
  - 管理后台：移动端功能优先隐藏，保留核心审核操作
- **触摸优化**：按钮最小触控面积 44×44px，删除操作需二次确认

## 4. 安全设计

### 4.1 认证与授权流程

```
┌──────────┐         ┌──────────┐         ┌──────────┐
│  Client  │         │ FastAPI  │         │PostgreSQL│
└────┬─────┘         └────┬─────┘         └────┬─────┘
     │   POST /auth/login  │                    │
     │   {user, password}  │                    │
     │────────────────────>│                    │
     │                     │  SELECT user       │
     │                     │───────────────────>│
     │                     │  user + hash       │
     │                     │<───────────────────│
     │                     │                    │
     │                     │  bcrypt.verify()   │
      │                     │  生成 JWT (HS256)  │
      │                     │  access: 1h        │
      │                     │  refresh: 7d       │
     │                     │                    │
     │   {access_token}    │                    │
     │<────────────────────│                    │
     │                     │                    │
     │   GET /api/v1/xxx   │                    │
     │   Authorization:    │                    │
     │   Bearer <JWT>      │                    │
     │────────────────────>│                    │
     │                     │  验证 JWT 签名      │
     │                     │  检查 exp           │
     │                     │  提取 role          │
     │                     │  检查接口权限        │
     │                     │                    │
     │   Response          │                    │
     │<────────────────────│                    │
```

### 4.3 安全措施汇总

| 层面 | 措施 | 实现方式 |
|------|------|----------|

| **文件安全** | 类型白名单 + 大小限制 | SecurityService 校验 |
| **CSRF** | SameSite Cookie + JWT Header | 不依赖 Cookie，JWT 放 Header |
| **CORS** | 白名单跨域策略 | FastAPI CORSMiddleware |
| **日志** | 关键操作全记录 | audit_logs 表 + loguru 文件日志 |

## 5. 部署方案

> **说明**：Docker 为可选方案。Windows 环境下已配置 WSL2，可直接使用 Docker Desktop。如不使用 Docker，PostgreSQL 和 ChromaDB 均可本地安装（`pip install chromadb` 即可启动 ChromaDB）。

### 5.1 Docker Compose 配置（可选）

```yaml
# docker-compose.yml (Docker Compose v2，无需 version 字段)

services:
  db:
    image: postgres:16-alpine
    container_name: edurag-db
    environment:
      POSTGRES_DB: edurag
      POSTGRES_USER: edurag
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data
      - ./scripts/backup:/backup      # pg_dump 备份目录
    healthcheck:                       # 确保 backend 启动前 db 已就绪
      test: ["CMD-SHELL", "pg_isready -U edurag"]
      interval: 5s
      timeout: 5s
      retries: 5
    restart: unless-stopped

  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    container_name: edurag-backend
    environment:
      DATABASE_URL: postgresql+asyncpg://edurag:${DB_PASSWORD}@db:5432/edurag
      LLM_API_KEY: ${LLM_API_KEY}
      LLM_BASE_URL: ${LLM_BASE_URL:-https://api.deepseek.com}
      LLM_MODEL: ${LLM_MODEL:-deepseek-v4-flash}
      EMBEDDING_PROVIDER: ${EMBEDDING_PROVIDER:-siliconflow}
      EMBEDDING_MODEL: ${EMBEDDING_MODEL:-BAAI/bge-m3}
      SILICONFLOW_API_KEY: ${SILICONFLOW_API_KEY}
    ports:
      - "8000:8000"
    volumes:
      - ./data/files:/app/data/files
      - ./data/chroma:/app/data/chroma  # ChromaDB 嵌入式运行，无需独立容器
    depends_on:
      db:
        condition: service_healthy      # 等待 db healthcheck 通过
    restart: unless-stopped

  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile
    container_name: edurag-frontend
    ports:
      - "3000:80"
    depends_on:
      - backend
    restart: unless-stopped

volumes:
  pgdata:
```

> **关于 ChromaDB**：本项目使用 `chromadb.PersistentClient`（嵌入式模式），数据持久化到 `./data/chroma` 目录。ChromaDB 直接运行在 backend 容器内，无需独立容器。生产环境如需扩展，可部署 ChromaDB Server 作为独立服务。

> **关于前端 Nginx**：前端的 `Dockerfile` 需基于 Nginx 镜像，并配置 SPA 路由 fallback（所有非 `/api/` 请求返回 `index.html`）。示例 `nginx.conf`：
> ```
> location / {
>     try_files $uri $uri/ /index.html;
> }
> location /api/ {
>     proxy_pass http://backend:8000;
> }
> ```

### 5.5 错误处理与日志规范

**统一错误码体系**：

| 错误码段 | 含义 | 示例 |
|----------|------|------|
| 0 | 成功 | `{"code": 0}` |
| 40001-40099 | 参数校验错误 | 40001 用户名已存在、40002 参数缺失 |
| 40100-40199 | 认证/授权错误 | 40100 Token 过期、40101 无权限 |
| 40300-40399 | 业务逻辑拒绝 | 40300 文件类型不支持、40301 触发安全策略 |
| 50000-50099 | 服务端内部错误 | 50000 模型加载失败、50001 LLM API 超时 |

**FastAPI 统一异常处理中间件**：

```python
# backend/app/middleware/error_handler.py
import traceback
from fastapi import Request, HTTPException
from fastapi.responses import JSONResponse
from fastapi.exceptions import RequestValidationError
from sqlalchemy.exc import IntegrityError
from loguru import logger

async def http_exception_handler(request: Request, exc: HTTPException):
    """HTTP 异常（401/403/404 等）"""
    return JSONResponse(
        status_code=exc.status_code,
        content={"code": 40000 + exc.status_code, "message": exc.detail, "data": None}
    )

async def validation_exception_handler(request: Request, exc: RequestValidationError):
    """Pydantic 参数校验失败"""
    errors = [{"loc": e["loc"], "msg": e["msg"]} for e in exc.errors()]
    logger.warning(f"Validation error: {errors}")
    return JSONResponse(
        status_code=422,
        content={"code": 40002, "message": "参数校验失败", "data": {"errors": errors}}
    )

async def integrity_error_handler(request: Request, exc: IntegrityError):
    """数据库约束冲突（如 UNIQUE 重复）"""
    logger.error(f"Integrity error: {exc}")
    return JSONResponse(
        status_code=409,
        content={"code": 40302, "message": "数据冲突，操作无法完成", "data": None}
    )

async def global_exception_handler(request: Request, exc: Exception):
    """全局兜底异常捕获"""
    logger.error(f"Unhandled exception: {traceback.format_exc()}")
    return JSONResponse(
        status_code=500,
        content={"code": 50000, "message": "服务器内部错误", "data": None}
    )
```

**日志级别策略（loguru）**：

| 级别 | 记录内容 |
|------|----------|
| DEBUG | SQL 执行详情、Embedding 向量维度 |
| INFO | 用户登录/登出、文档上传、问答请求、审核操作 |
| WARNING | 查询无结果、相似度过低、文件格式不支持、限流触发 |
| ERROR | LLM API 调用失败、模型加载异常、数据库连接断开 |
| CRITICAL | 服务无法启动、磁盘空间不足 |
