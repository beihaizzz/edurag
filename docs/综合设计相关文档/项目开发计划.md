## 1. 项目概述

### 1.1 项目定义

面向学生和教师，构建一个课程资料上传、检索、问答和反馈的一体化服务平台。系统将分散存放的课件（PPT/PDF）、实验指导书、作业要求等教学资料集中管理，并通过 RAG（检索增强生成）技术提供基于自然语言的智能问答。

### 1.2 核心价值

| 价值点 | 说明 |
|--------|------|
| 资料集中化 | 解决"课件在哪""作业要求是什么"等高频重复问题 |
| 智能检索 | 关键词精确匹配 + 向量语义检索双模式 |
| RAG 问答 | 基于知识库文档生成答案，标注引用来源，杜绝编造 |
| 安全可控 | 管理端审核 + 提示注入防护 + 无关问题拒答 |
| 反馈闭环 | 记录用户反馈，持续优化检索与生成质量 |

### 1.3 功能全览

| 编号 | 功能模块 | 类别 | 角色 |
|:---:|---------|:---:|------|

## 2. 技术架构确认

### 2.1 技术栈

| 层级 | 选型 | 说明 |
|------|------|------|
| **后端框架** | Python 3.10+ / FastAPI + Uvicorn | 异步高性能，原生支持 LLM/AI 生态 |
| **前端框架** | React 18 + TypeScript + Vite | SPA（学生/教师端）+ 独立管理后台 SPA |
| **关系数据库** | PostgreSQL 15+ + SQLAlchemy 2.0 ORM | 存储元数据、用户、问答历史 |
| **向量数据库** | ChromaDB (PersistentClient) | 存储 chunk 向量，余弦相似度检索 |
| **缓存** | Redis（可选，初期可省略） | Token 黑名单/限流（非硬性依赖） |
| **文档解析** | Docling（主解析）+ PyMuPDF（兜底） | PDF/DOCX/PPTX/TXT/图片 |
| **Chunk 切分** | LangChain RecursiveCharacterTextSplitter | 中文适配分隔符，保持语义连续 |
| **Embedding** | BAAI/bge-m3 (1024d) via SiliconFlow API（推荐）/ bge-large-zh-v1.5 本地兜底 | 双 Provider 抽象，生产用 API 免 GPU，离线降级本地模型 |
| **LLM API** | DeepSeek v4-flash | 云端 API，temperature=0.3，max_tokens=1024 |
| **RAG 编排** | LangChain LCEL | ChatPromptTemplate → ChatDeepSeek → StrOutputParser |
| **包管理** | uv (pyproject.toml) / pnpm (package.json) | 快速依赖管理 |
| **数据库迁移** | Alembic | 版本化 schema 变更 |

### 2.2 架构分层

```
展示层     →  SPA（学生/教师端）+ Admin SPA（管理后台）
接口层     →  FastAPI（Auth/CORS/限流/日志中间件）
业务层     →  Service 层（User/Document/Search/QA/Feedback/Admin/Security）
数据层     →  PostgreSQL + ChromaDB + 本地文件存储 (data/files/)
```

### 6.3 RAG 管线技术细节

```
用户问题
  │
  ├─ 1. QAService._check_input_safety(question)
  │     → 正则匹配注入模式 → 命中则拒绝
  │
  ├─ 2. 混合检索（asyncio.gather 并行）
  │     ├─ semantic_search: 问题向量化 → ChromaDB.query(query_embeddings, n_results=10)
  │     └─ keyword_search: 关键词提取 → ILIKE 模糊匹配 chunks.content (Top-5)
  │     → 去重合并 → 按分数降序排列
  │
  ├─ 3. 相关性过滤
  │     → cosine_similarity < 0.45 的片段淘汰
  │     → 全部淘汰？→ _build_rejection() 拒答
  │
  ├─ 4. Prompt 构建
  │     └─ System Prompt（角色约束 + 安全规则 + 引用格式要求）
  │        + 参考资料列表（[1], [2], [3]...）
  │        + 用户问题
  │
  ├─ 5. LLM 生成
  │     ├─ stream=false: chain.invoke() → 等待完整返回
  │     └─ stream=true:  chain.astream() → SSE StreamingResponse
  │
  └─ 6. 结果处理
        ├─ 保存到 qa_history（user_id, question, answer, sources JSON, course_id, latency_ms）
        ├─ 校验输出是否含引用标注 → 不含则触发拒答
        └─ 返回 { answer, sources, is_rejected }
```

**提交规范**：`type(scope): description`
- `feat(backend): implement user registration API`
- `fix(search): correct ILIKE escape handling`

### 8.2 代码规范

- **Python**: PEP 8，类型注解（Pydantic models），async/await
- **TypeScript**: strict mode，interface 优先于 type
- **命名**: 后端 snake_case，前端 camelCase，数据库 snake_case
- **错误处理**: 绝不使用 `try: ... except: pass`，必须日志记录
- **禁止**: `as any`、`@ts-ignore`、裸 `except`、硬编码密钥

## 9. 风险评估与对策

### 9.1 技术风险

| 风险 | 影响 | 概率 | 对策 |
|------|:---:|:---:|------|
| **SiliconFlow API 不可用 / 限流** | 阻塞向量化全部流程 | 低 | 自动降级到本地 bge-large-zh-v1.5 模型；或使用 HuggingFace 国内镜像下载本地模型备用 |
| **DeepSeek API 限流/不可用** | 阻塞 RAG 问答 | 低 | 申请多 Key 轮换；备选通义千问 API（qwen-turbo）；准备本地 Qwen2.5-1.5B 小模型兜底 |
| **Docling 解析复杂 PDF 表格错乱** | 检索质量下降 | 中 | PyMuPDF 兜底；对解析结果做人工抽查；错乱表格不影响关键词匹配 |
| **ChromaDB 向量维度不匹配** | 检索报错 | 低 | 建表时固定维度 1024；模型更换前需清空旧向量并重新入库 |
| **本地 Embedding 推理慢（仅 EMBEDDING_PROVIDER=local 时）** | 仅影响离线环境 | 低 | 本地 bge-large-zh 在 CPU 上速度约 10-20 docs/min；生产推荐使用 SiliconFlow API（<10ms/batch） |
| **前端 SSE 流式响应在代理/网关下被缓冲** | 流式问答变成一次性返回 | 低 | Nginx 配置 `proxy_buffering off`；FastAPI 端设置 `X-Accel-Buffering: no` 头 |

### 9.2 进度风险

| 风险 | 影响 | 概率 | 对策 |
|------|:---:|:---:|------|
| **第 1 周环境搭建耗时超预期** | 压缩后续开发时间 | 中 | Docker 一键启动 PostgreSQL+ChromaDB；uv 快速安装 Python 依赖；提前准备 .env 模板 |
| **RAG 管线调试时间不足（F8/F9 复杂度高）** | 第 3 周无法完成核心功能 | 高 | 优先走通最小 RAG 管线（解析→单 chunk→直接 LLM），再逐步优化切分和检索策略 |
| **前端页面数量多，开发时间紧张** | 第 4 周赶工质量差 | 中 | 使用组件库（Ant Design）减少手写代码；优先学生端核心页面，教师端/管理端可简化 |
| **课程资料收集不足/质量差** | 演示效果差 | 低 | 优先选择资料丰富的课程（如数据结构、操作系统）；利用公开课程网站补充 |
