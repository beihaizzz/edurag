## 1. 项目背景与意义

### 1.1 背景

在日常教学活动中，每门课程都会产生大量教学资料——课件（PPT/PDF）、实验指导书、作业要求文档、参考资料等。这些资料通常分散存放在不同平台（如QQ群文件、学习通、教师个人主页、微信群等），学生查找时往往需要反复翻找、多次询问，效率低下，甚至可能遗漏关键信息。

与此同时，大语言模型（LLM）和检索增强生成（RAG）技术的快速发展，使得基于自然语言的智能问答成为可能。将这些技术应用于课程资料管理，可以极大改善学生和教师的资料检索体验。

### 1.2 意义

- **解决痛点**：将分散的课程资料集中管理，解决"课件在哪""作业要求是什么"等高频重复问题
- **提升效率**：通过关键词检索 + 向量语义检索 + LLM 问答，学生可用自然语言快速定位所需信息
- **可追溯性**：问答历史、反馈记录等机制确保信息可追溯，持续优化系统质量
- **安全可控**：通过管理端审核、提示注入防护、无答案拒答等机制，确保系统输出安全可靠
- **实践价值**：项目涵盖 Web 开发、数据库设计、文本处理、信息检索、LLM 调用等核心技能，有较高的综合实践价值

## 2. 项目目标

面向学生和教师，构建一个课程资料上传、检索、问答和反馈的一体化服务平台。系统应支持：

1. **资料集中管理**：教师/管理员上传课件、实验指导书、作业要求等文档，系统自动解析、分块、向量化入库
2. **多模式检索**：支持关键词精确检索与语义模糊检索，快速定位相关文档和片段
3. **智能问答**：基于 RAG 架构，结合检索到的文档片段与 LLM 生成自然语言答案，并明确标注引用来源
4. **安全机制**：管理端审核、提示注入防护、无相关答案时拒答（不编造内容）
5. **用户反馈闭环**：记录问答历史，支持用户对答案进行反馈（有用/无用/错误），持续优化检索与生成质量

## 4. 功能需求

| **输入** | 分类筛选条件；元数据编辑表单 |

| **输入** | 无（自动记录） |
| **输出** | 问答历史列表 |

#### 4.1.6 反馈记录模块

| 项目 | 内容 |

| 项目 | 内容 |

| **详细要求** | (1) 审核列表展示待审核文档及元数据；(2) 支持通过/驳回操作，驳回需填写原因；(3) 审核通过后文档正式入库可被检索；(4) 记录审核日志 |
| **输入** | 审核决策（通过/驳回 + 原因） |
| **输出** | 审核状态变更；日志记录 |

### 4.2 前沿增强模块

#### 4.2.1 RAG 文档切分与向量检索

| 项目 | 内容 |
|------|------|
| **功能描述** | 将文档切分为语义块（chunk），向量化后存入向量数据库，实现语义检索 |
| **详细要求** | (1) 文档解析：提取 PDF/DOCX/TXT 等文本内容；(2) Chunk 切分：按语义段落切分，保持上下文连贯（建议 chunk_size: 500-1000 tokens, overlap: 100-200 tokens）；(3) 向量化：调用 Embedding 模型（如 text2vec、bge-large-zh 等）生成向量；(4) 向量入库：存入向量数据库（如 ChromaDB、Milvus、FAISS）；(5) 语义检索：用户问题向量化后，在向量库中进行相似度检索，返回 Top-K 相关片段 |
| **输入** | 文档文件 / 用户自然语言问题 |
| **输出** | 向量化 chunk 列表 / Top-K 相关文档片段 + 相似度分数 |

#### 4.2.2 LLM 答案生成与来源引用

| 项目 | 内容 |
|------|------|
| **功能描述** | 基于检索到的文档片段，调用 LLM 生成自然语言答案并标注引用 |
| **详细要求** | (1) Prompt 构建：将用户问题 + 检索到的 Top-K 片段拼接为上下文 prompt；(2) LLM 调用：调用本地小模型（如 ChatGLM、Qwen）或云端 API（如 DeepSeek、通义千问）；(3) 答案格式：自然语言回答 + 引用编号标注（如 [1] [2]）；(4) 引用展示：前端展示答案时同时展示引用片段原文与来源文档 |
| **输入** | 用户问题 + 检索到的文档片段 |
| **输出** | 自然语言答案 + 引用标注 + 来源文档链接 |

#### 4.2.3 提示注入防护

| 项目 | 内容 |
|------|------|
| **功能描述** | 防止恶意用户通过构造特殊输入操控 LLM 行为 |
| **详细要求** | (1) 输入过滤：检测并过滤常见的提示注入模式（如 "ignore previous instructions"、"system:" 等）；(2) 输出约束：LLM 输出仅基于提供的文档上下文，不允许执行用户嵌入的指令；(3) Prompt 加固：在系统 prompt 中明确限制 LLM 角色与行为边界 |
| **输入** | 用户原始输入 |
| **输出** | 清洗后的安全输入 / 拦截提示 |

#### 4.2.4 无答案拒答

| 项目 | 内容 |
|------|------|
| **功能描述** | 当检索结果相关度不足时，系统拒答而非编造答案 |
| **详细要求** | (1) 相关性阈值：设定相似度阈值（如 0.6），低于阈值视为无关；(2) 拒答策略：当 Top-K 片段均低于阈值时，返回 "未找到相关资料，无法回答该问题"，并建议用户尝试其他关键词或联系教师；(3) 日志记录：拒答事件记录日志，便于后续分析知识库覆盖盲区 |
| **输入** | Top-K 文档片段 + 相似度分数 |
| **输出** | 正常答案 或 拒答提示 |

## 5. 非功能性需求

### 5.1 性能要求

| 指标 | 要求 |
|------|------|
| 检索响应时间 | 关键词检索 < 1s，语义检索 < 3s |
| LLM 问答响应时间 | < 10s（本地小模型）/< 5s（云端 API） |
| 文档解析时间 | 单文件 < 5s |
| 并发支持 | 至少支持 20 并发用户 |

### 5.2 安全要求

| 要求 | 说明 |
|------|------|
| 身份认证 | JWT / Session 机制，密码加密存储（bcrypt） |
| 权限控制 | 角色级权限（学生/教师/管理员），越权操作拦截 |
| 输入安全 | XSS 过滤、SQL 注入防护、文件上传类型校验 |
| 提示注入防护 | 输入过滤 + Prompt 加固（见 4.2.3） |
| 数据脱敏 | 上传资料中如含学生个人信息，需提示脱敏处理 |

### 5.3 可用性要求

| 要求 | 说明 |
|------|------|
| 界面简洁 | 学生端以搜索框为核心入口，操作直观 |
| 移动适配 | 前端需支持移动端浏览器基本浏览 |
| 错误提示 | 操作失败时提供清晰的错误原因与建议 |

### 5.4 可维护性要求

| 要求 | 说明 |
|------|------|
| 日志记录 | 关键操作（上传、审核、问答、反馈）需记录日志 |
| 版本管理 | 代码使用 Git 管理，文档定期更新 |
| 模块化 | 前后端分离，各模块功能独立、接口清晰 |

## 6. 技术选型建议

| 层级 | 推荐方案 | 备选方案 | 说明 |
|------|----------|----------|------|
| **后端框架** | Python + FastAPI | Java + Spring Boot | FastAPI 生态对 LLM/AI 支持更好，开发效率高 |
| **前端框架** | React + TypeScript | Vue 3 + TypeScript | 任一均可，Vue 上手更快 |
| **数据库** | PostgreSQL + ChromaDB | MySQL + FAISS | 关系型存储元数据，向量库存储 embedding |
| **LLM** | DeepSeek API / Qwen | ChatGLM 本地部署 | 云端 API 省去 GPU 环境搭建，答案质量更稳定 |
| **Embedding** | bge-large-zh-v1.5 | text2vec-large-chinese | 中文语义向量效果优秀 |
| **文档解析** | Docling + PyMuPDF（兜底） | LangChain Document Loaders | Docling 主解析多格式（表格精度 97.9%），PyMuPDF 纯文本兜底 |
| **Chunk 切分** | LangChain TextSplitter | 自实现递归切分 | 成熟方案，参数灵活 |

```
用户表 (users)
```

### 核心业务流程

**文档入库流程**:
```
教师上传文档 → 填写元数据 → 管理员审核 → 通过后：
  → 文件存储 (本地/OSS)
  → 保存元数据 (PostgreSQL)
  → 文档解析 (PDF/DOCX → 纯文本)
  → Chunk 切分 (语义段落)
  → 向量化 (Embedding Model)
  → 向量入库 (ChromaDB)
```

**智能问答流程**:
```
用户提问 → 输入安全检查 → 问题向量化
  → 向量检索 (ChromaDB Top-K)
  → 相关性判断 → 低于阈值 → 拒答
               → 高于阈值 → 构建 Prompt
                 → LLM 生成答案
                 → 答案 + 引用标注
  → 记录问答历史 → 展示答案
```
